# Fase 3: Sistema de Handshake para Produção - Requisitos

## Visão Geral

O Sistema de Handshake para Produção permite que alterações em pedidos que já estão em produção sejam comunicadas em tempo real para os operadores, que podem aprovar ou rejeitar essas mudanças antes que sejam aplicadas.

## Objetivos

1. **Comunicação em Tempo Real**: Notificar operadores imediatamente sobre solicitações de alteração
2. **Controle de Qualidade**: Permitir que operadores avaliem o impacto das alterações na produção
3. **Rastreabilidade**: Manter histórico completo de todas as alterações solicitadas e aprovadas
4. **Flexibilidade Operacional**: Permitir alterações necessárias sem interromper a produção

## Requisitos Funcionais

### RF01 - Sistema de Notificações WebSocket
- O sistema deve implementar comunicação WebSocket para notificações em tempo real
- Deve suportar salas por organização para isolamento de dados
- Deve reconectar automaticamente em caso de perda de conexão
- Deve manter histórico de notificações não lidas

### RF02 - Fluxo de Solicitação de Alteração
- Quando um pedido em status IN_PRODUCTION for alterado, o sistema deve:
  - Salvar as alterações em uma tabela `PendingChanges`
  - Enviar notificação WebSocket para operadores da organização
  - Manter o pedido no estado atual até aprovação
  - Mostrar indicador visual de "alteração pendente"

### RF03 - Interface de Aprovação para Operadores
- Painel dedicado para operadores visualizarem solicitações pendentes
- Detalhes da alteração (antes/depois) de forma clara
- Botões de Aprovar/Rejeitar com comentários opcionais
- Filtros por produto, urgência, data
- Notificações sonoras/visuais para novas solicitações

### RF04 - Aplicação de Alterações Aprovadas
- Após aprovação, aplicar automaticamente as alterações ao pedido
- Registrar no histórico do pedido quem aprovou e quando
- Notificar o solicitante sobre a aprovação/rejeição
- Remover da lista de pendências

### RF05 - Gestão de Permissões
- Apenas usuários com role OPERATOR ou ADMIN podem aprovar alterações
- Usuários SALES podem solicitar alterações
- Logs de auditoria para todas as ações

## Requisitos Não Funcionais

### RNF01 - Performance
- Notificações WebSocket devem ser entregues em < 100ms
- Interface deve suportar até 50 solicitações simultâneas
- Reconexão automática em < 5 segundos

### RNF02 - Usabilidade
- Interface intuitiva com indicadores visuais claros
- Notificações não intrusivas mas visíveis
- Responsiva para tablets usados na produção

### RNF03 - Confiabilidade
- Sistema deve funcionar mesmo com conexão instável
- Backup de notificações em caso de falha
- Recuperação automática de estado

## Casos de Uso

### UC01 - Solicitar Alteração em Pedido em Produção
**Ator**: Vendedor
**Pré-condições**: Pedido em status IN_PRODUCTION
**Fluxo Principal**:
1. Vendedor acessa pedido em produção
2. Faz alterações necessárias (quantidade, especificações, etc.)
3. Sistema detecta que pedido está em produção
4. Sistema salva alterações como pendentes
5. Sistema envia notificação para operadores
6. Sistema mostra status "Aguardando Aprovação" para vendedor

### UC02 - Aprovar/Rejeitar Alteração
**Ator**: Operador
**Pré-condições**: Existe alteração pendente
**Fluxo Principal**:
1. Operador recebe notificação de nova solicitação
2. Acessa painel de aprovações
3. Visualiza detalhes da alteração
4. Avalia impacto na produção
5. Aprova ou rejeita com comentário
6. Sistema aplica alteração (se aprovada) ou descarta (se rejeitada)
7. Sistema notifica solicitante do resultado

### UC03 - Monitorar Status de Solicitações
**Ator**: Vendedor/Gerente
**Pré-condições**: Existem solicitações em andamento
**Fluxo Principal**:
1. Usuário acessa dashboard de solicitações
2. Visualiza status de todas as solicitações
3. Pode filtrar por status, data, produto
4. Recebe notificações de mudanças de status

## Estrutura de Dados

### Tabela: PendingChanges
```sql
CREATE TABLE PendingChanges (
  id UUID PRIMARY KEY,
  orderId UUID NOT NULL,
  organizationId UUID NOT NULL,
  requestedBy UUID NOT NULL,
  requestedAt TIMESTAMP NOT NULL,
  changes JSONB NOT NULL, -- Alterações solicitadas
  originalData JSONB NOT NULL, -- Estado original
  status ENUM('PENDING', 'APPROVED', 'REJECTED') DEFAULT 'PENDING',
  reviewedBy UUID,
  reviewedAt TIMESTAMP,
  reviewComments TEXT,
  priority ENUM('LOW', 'MEDIUM', 'HIGH') DEFAULT 'MEDIUM',
  createdAt TIMESTAMP DEFAULT NOW(),
  updatedAt TIMESTAMP DEFAULT NOW()
);
```

### Tabela: Notifications
```sql
CREATE TABLE Notifications (
  id UUID PRIMARY KEY,
  organizationId UUID NOT NULL,
  userId UUID, -- NULL para notificação geral
  type ENUM('CHANGE_REQUEST', 'CHANGE_APPROVED', 'CHANGE_REJECTED'),
  title VARCHAR(255) NOT NULL,
  message TEXT NOT NULL,
  data JSONB, -- Dados específicos da notificação
  read BOOLEAN DEFAULT FALSE,
  createdAt TIMESTAMP DEFAULT NOW()
);
```

## Fluxos de Integração

### Integração com Sistema de Pedidos
- Interceptar alterações em pedidos IN_PRODUCTION
- Validar permissões antes de salvar como pendente
- Aplicar alterações após aprovação

### Integração com Sistema de Usuários
- Verificar roles para permissões de aprovação
- Identificar operadores para envio de notificações
- Registrar ações para auditoria

### Integração com Frontend
- Componentes de notificação em tempo real
- Interface de aprovação responsiva
- Indicadores visuais de status

## Critérios de Aceite

### Para Vendedores:
- ✅ Pode solicitar alterações em pedidos em produção
- ✅ Recebe feedback visual de que alteração está pendente
- ✅ É notificado quando alteração é aprovada/rejeitada
- ✅ Pode acompanhar status de suas solicitações

### Para Operadores:
- ✅ Recebe notificações em tempo real de novas solicitações
- ✅ Pode visualizar detalhes completos da alteração
- ✅ Pode aprovar/rejeitar com comentários
- ✅ Interface funciona bem em tablets de produção

### Para o Sistema:
- ✅ Mantém integridade dos dados durante todo o processo
- ✅ Registra auditoria completa de todas as ações
- ✅ Performance adequada mesmo com múltiplas solicitações
- ✅ Recupera automaticamente de falhas de conexão

## Riscos e Mitigações

### Risco: Perda de notificações por falha de conexão
**Mitigação**: Sistema de backup com polling periódico

### Risco: Sobrecarga de notificações
**Mitigação**: Agrupamento inteligente e filtros de prioridade

### Risco: Conflitos de alterações simultâneas
**Mitigação**: Versionamento otimista e detecção de conflitos

### Risco: Interface complexa para operadores
**Mitigação**: Design focado em usabilidade e testes com usuários reais
